# Court IQ v0.1 Release Notes & Known Limitations

## 1) Release summary

Court IQ v0.1 is a **local demo/reviewer build** that is ready to share for guided walkthroughs and internal evaluation. It aligns with recent review verdicts:

- R3: `DEMO_READY_WITH_CAVEATS`
- R5: `TRUST_READY_WITH_WARNINGS`
- R1: `READY_WITH_RISKS`

Combined release verdict for v0.1: **READY_WITH_RISKS**.

This release is intended to demonstrate end-to-end workflow coherence and explainable outputs on deterministic sample data, while making limitations explicit.

## 2) What v0.1 demonstrates

v0.1 demonstrates that Court IQ can:

- Run a local-first intake-to-artifact workflow across coach, analyst, and player contexts.
- Produce explainable, role-oriented outputs from deterministic sample project data.
- Support structured review handoffs and reproducible demo scenarios.
- Generate local Player Value outputs with interpretable reasoning traces.

## 3) Recommended demo flow

Use the following reviewer flow for consistent demos:

1. Start backend and frontend locally.
2. Seed deterministic sample data.
3. Open the Home page and load sample project context.
4. Walk reviewer through coach workflow outputs.
5. Walk reviewer through analyst review/calibration touchpoints.
6. Show player-facing training/decision artifacts.
7. Review trust/scoring warnings before discussing interpretation.
8. Close with known limitations and next fixes.

For a scripted walkthrough, use: `docs/product/demo-script.md`.

## 4) Supported workflows

v0.1 supports these documented workflows:

- Coach workflow (`docs/workflows/coach-workflow.md`)
- Analyst workflow (`docs/workflows/analyst-workflow.md`)
- Player workflow (`docs/workflows/player-workflow.md`)

These are suitable for demo/reviewer use with deterministic inputs.

## 5) Known limitations

- This is **not** a production release; it is a **local demo/reviewer build**.
- Player identity is **alias-based**, not real-world automatic recognition.
- Sample project behavior is tied to deterministic seeded data and may not represent live variability.
- Trust/scoring indicators include warning states that require human review.
- Output quality depends on the sample flow and expected reviewer path.
- Robustness outside documented flows is limited in v0.1.

## 6) Trust / scoring caveats

- **Trust warnings should be reviewed before interpreting results.**
- Trust and scoring outputs are decision-support signals, not standalone truth.
- Reviewer judgment is required when warnings, missing context, or edge cases appear.
- Comparisons between players/projects should be treated as local-demo directional guidance only.

## 7) Data / model caveats

- The sample project uses **deterministic sample data** for reproducibility.
- Local scoring/model behavior is tuned for explainability and workflow clarity in demo conditions.
- Real-world distribution shifts, noisy inputs, and incomplete records are not fully represented in v0.1.
- Reproducibility is prioritized over broad real-world coverage in this release.

## 8) What not to claim

Do **not** claim that v0.1:

- Is production-ready or validated for autonomous decisions.
- Performs real-world automatic player identity recognition.
- Produces official scouting grades.
- Provides independent, final decision authority without human review.

Required positioning statement:

- **Player Value is explainable and local, not an official scouting grade.**

## 9) How to run the sample project

From repo root, run backend and frontend:

```bash
# Terminal 1
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

```bash
# Terminal 2
cd frontend
npm install
npm run dev
```

Seed deterministic sample data:

```bash
curl http://localhost:8000/api/sample-data/status
curl -X POST http://localhost:8000/api/sample-data/seed
```

Open: <http://localhost:5173>
