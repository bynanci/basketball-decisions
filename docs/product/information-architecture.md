# Product Information Architecture

Issue #81 groups the existing frontend surface area by job-to-be-done while keeping the current routes available. The Development Dashboard is the main command center because it links operational health, next-best-actions, and this grouped product map without changing backend APIs or adding analytics features.

## Top-level product groups

| Group | Purpose | Primary entry points |
| --- | --- | --- |
| Analyze | Turn video projects into calibrated tracking, decisions, and coach-facing reporting artifacts. | Project Workspace, Calibration, Pipeline, Tracking, Coach Reports |
| Review | Queue, inspect, and resolve human review work generated by tracking and source artifacts. | Review Queue, Tracking Review, Reference Videos |
| Player Value | Build, explain, and inspect local Player Value summaries and trends. | Player Value Summary, Player Value Trends, Player Value Evidence |
| Training | Role-based learning, decision quizzes, drills, and practice planning flows. | Choose Training Role, Training Lobby, Situations, Quiz Builder, Drills, Practice Plans, Practice Executions |
| System / Lab | Command center, orchestration, source setup, rules, model registry, and local lab operations. | Development Dashboard, Home / Intake, Workflows, Local Lab, Model Registry, Decision Rules |

## Navigation rules

- The topbar points the brand and first action at `/development-dashboard` to make the Development Dashboard the command center.
- Static, user-enterable routes are grouped under visible section labels in the topbar.
- Dynamic routes stay available through their parent/context pages and in this inventory, but they are not primary topbar links because they require project, player, workflow, prompt, execution, or reference IDs.
- Existing routes, route names, and backend API contracts are unchanged.
